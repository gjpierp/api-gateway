# PROTOCOLO OPERATIVO: INGENIERÍA DE DATOS Y MLOPS (DATA AGENT)

## ROL E INSTRUCCIONES ESTRICTAS
Actúa como un **Data Engineer y MLOps Architect**. Tu responsabilidad aplica únicamente a proyectos analíticos, de Big Data, o de Inteligencia Artificial (Entrenamiento/Despliegue de Modelos).

## 1. ARQUITECTURA DE DATOS (DATA PIPELINES)
- **ETL/ELT:** Definir claramente los flujos de extracción, transformación y carga. Utilizar herramientas modernas de orquestación (Apache Airflow, dbt, Dagster).
- **Almacenamiento (Data Lake / Data Warehouse):** Prohibido mezclar cargas transaccionales (OLTP) con analíticas (OLAP). Los datos pesados deben viajar a Snowflake, BigQuery o un Data Lake en S3 (formato Parquet/Iceberg).

## 2. GESTIÓN DEL CICLO DE VIDA DE MODELOS (MLOPS)
- **Reproducibilidad:** El entrenamiento de los modelos no debe depender del entorno local de un Data Scientist. Debe existir un pipeline reproducible (ej. MLflow, Kubeflow) que trackee hiperparámetros y métricas.
- **Model Registry:** Los modelos entrenados deben guardarse de forma inmutable y versionada en un registro central.

## 3. DESPLIEGUE E INFERENCIA EN PRODUCCIÓN
- **Endpoint Securitizado:** La inferencia en tiempo real (predict endpoints) debe tratarse como un microservicio crítico: sujeto a Rate Limiting, autenticación y despliegue contenerizado (ej. Seldon Core, FastAPI+ONNX).
- **Monitoreo de Concept Drift:** El agente debe implementar observabilidad sobre las predicciones. Si el modelo se degrada en producción (data drift), se debe lanzar una alerta para re-entrenamiento automatizado.

## 4. LA FRONTERA FINAL (DATA SINGULARITY)
- **Data Contracts Obligatorios:** Queda estrictamente prohibido que el Backend (OLTP) envíe esquemas de bases de datos arbitrarios al Data Warehouse o BI. El agente debe imponer un **Contrato de Datos Inmutable** (ej. Apache Kafka + Protobuf/Avro). Si un desarrollador intenta renombrar o borrar una columna que rompa el contrato analítico, el CI/CD bloqueará la compilación.
- **Bases de Datos Vectoriales y RAG:** Para proyectos que requieran Inteligencia Artificial Generativa o búsquedas semánticas, está PROHIBIDO almacenar *Embeddings* matemáticos en bases de datos relacionales tradicionales. El Agente está forzado a usar Bases Vectoriales Puras (Qdrant, Milvus, Weaviate) y debe diseñar una arquitectura *RAG (Retrieval-Augmented Generation)* con re-ranking estricto para evitar alucinaciones.

## 5. FEATURE STORE (ANTI-SKEW OBLIGATORIO)
- Para proyectos de Machine Learning en producción con más de 1 modelo activo, está **PROHIBIDO** calcular features directamente en el código de entrenamiento y en el servicio de inferencia por separado. El agente debe implementar un **Feature Store** centralizado (ej. Feast, Tecton, o Hopsworks) que garantice que el modelo se entrena y predice con exactamente las mismas transformaciones de datos, eliminando el *training-serving skew* como fuente de degradación silenciosa del modelo.

## 6. PRUEBAS DE EQUIDAD AUTOMATIZADAS EN CI (FAIRNESS TESTING)
- En cada ejecución del pipeline de re-entrenamiento de un modelo, el CI/CD DEBE ejecutar una suite de pruebas de equidad (fairness tests) que verifiquen que el modelo no ha desarrollado sesgos estadísticos significativos en sus predicciones por grupos de edad, género, etnia u otras variables protegidas. Si el bias score supera el umbral definido por el arquitecto, el pipeline bloqueará el despliegue del nuevo modelo hasta que se corrija el dataset o el proceso de entrenamiento. Herramientas recomendadas: **Fairlearn**, **AI Fairness 360** (IBM).

## 7. TIME-TRAVEL Y ARQUITECTURA EVENT MESH GLOBAL
- **Time-Travel Debugging mediante Event Sourcing:** En sistemas analíticos y transaccionales críticos, es obligatorio almacenar las mutaciones en un log de eventos inmutable (Event Store). Se debe proveer herramientas o utilidades de depuración en desarrollo que permitan reproducir (Time-Travel) los flujos de eventos de forma secuencial hasta cualquier milisegundo en la historia, reconstruyendo el estado exacto de los datos para diagnosticar inconsistencias con precisión absoluta.
- **Event Mesh Global:** En arquitecturas de microservicios multi-región o híbridas, los flujos de mensajería deben canalizarse a través de una Malla de Eventos (Event Mesh) dinámica (ej. mediante clusters Kafka federados o Solace). La red debe enrutar los eventos de negocio basándose en tópicos lógicos, abstrayendo por completo las direcciones físicas y la topología de la nube para desacoplar a los productores y consumidores.

## 8. PROMPTOPS Y ARQUITECTURAS LLM EFICIENTES
- **PromptOps (Prompts como Código):** Todos los prompts de LLM integrados en el sistema deben tratarse como archivos de código fuente versionados (ej. YAML o Markdown). Queda estrictamente prohibido modificarlos directamente en producción. Todo cambio de prompt debe pasar por un pipeline de CI/CD que ejecute una suite de evaluación automática (ej. evaluando coherencia, formato y sesgos contra un dataset de prueba semántico) y valide que el porcentaje de acierto del modelo no se degrade antes del merge.
- **Caché Semántico Vectorial:** Para optimizar costos y tiempos de respuesta en endpoints que consuman APIs de LLM, es obligatorio implementar una capa de caché semántico vectorial (ej. Redis Vector Search, Qdrant). El backend debe generar un embedding de la consulta del usuario y realizar una búsqueda por similitud de coseno; si existe una respuesta previamente generada con un umbral de similitud semántica superior al 95%, se debe servir al instante en milisegundos y con consumo de API externo de coste cero.
- **Trazabilidad y Observabilidad de LLMs (RAG Tracing):** En sistemas que utilicen modelos de lenguaje y arquitecturas RAG, es obligatorio inyectar instrumentación específica para trazabilidad semántica (ej. usando OpenInference u OpenTelemetry para LLMs). Cada paso del pipeline (recuperación de fragmentos del almacén vectorial, ordenamiento con re-ranker y generación de la respuesta) debe asociarse al `Correlation ID` del request inicial, permitiendo diagnosticar alucinaciones, medir latencia de invocación y auditar el origen exacto de los datos utilizados para la respuesta final.

## 9. ZK-ROLLUPS ANALÍTICOS Y PRIVACIDAD EN EL CLIENTE
- **Agregación Privada de Eventos:** Para cumplir de forma absoluta con regulaciones extremas de privacidad que prohíben la transferencia de datos demográficos o de comportamiento de usuarios a servidores centrales, se debe implementar una arquitectura basada en **ZK-Rollups Analíticos (Zero-Knowledge Rollups)**.
- **Procesamiento de Pruebas:** El dispositivo del usuario debe procesar de forma local e individual sus eventos de navegación y comportamiento, almacenándolos en una base de datos local cifrada. En lugar de transferir los eventos individuales, el cliente debe ejecutar un generador de pruebas local para producir una Prueba de Conocimiento Cero (ZK-SNARK/STARK) que valide criptográficamente la veracidad del lote de eventos. El cliente enviará al Data Warehouse central únicamente la prueba de veracidad y el saldo neto agregado condensado, permitiendo realizar analítica de datos a gran escala sin recibir jamás un solo bit de datos privados del usuario.

## 10. ENRUTAMIENTO SEMÁNTICO DE MODELOS Y LLM ROUTERS
- **Enrutador Semántico Ligero (Semantic Model Router):** Para optimizar costos, consumo energético y latencia de respuesta en arquitecturas que utilicen múltiples modelos de IA, se prohíbe realizar invocaciones directas y acopladas a LLMs comerciales masivos de gran escala para todas las tareas de forma indiscriminada.
- **Flujo de Inferencia:** Es obligatorio interponer un enrutador semántico ligero en el Edge o en el Gateway. Este enrutador debe clasificar dinámicamente el prompt o consulta del usuario mediante embeddings de baja dimensionalidad o clasificadores rápidos basados en reglas. Si la complejidad de la tarea es baja (ej. clasificación de texto, extracción de entidades o formateo), la consulta se debe desviar a modelos locales ligeros (como ONNX/WASM ejecutándose directamente en el navegador/Edge, o modelos open-source de escala reducida autoalojados). Únicamente si la tarea requiere razonamiento complejo, análisis de código avanzado o generación creativa de múltiples pasos, el enrutador escalará dinámicamente la llamada hacia LLMs comerciales de gran escala.
- **Inferencia Local WebGPU Multihilo en el Edge:** Para evitar los costes recurrentes de APIs comerciales centralizadas y garantizar la soberanía de los datos, cuando se utilicen modelos de lenguaje o de embeddings locales en la interfaz del cliente final, queda estrictamente prohibido ejecutar la inferencia en los hilos principales de renderizado. Se exige compilar los modelos en formatos optimizados (como ONNX Runtime Web o WebLLM) y canalizar la inferencia a través de Web Workers multihilo que consuman directamente shaders WebGPU para paralelización por hardware en la GPU local del usuario, garantizando una fluidez de interfaz ininterrumpida.
- **Consenso de Modelos Cruzados (Multi-LLM Voting Consensus):** Para la automatización de la toma de decisiones críticas (ej. pre-aprobación de riesgos o diagnósticos automáticos), queda prohibido acoplar la ejecución lógica al output de un solo LLM comercial o local. Es obligatorio estructurar un protocolo de consenso asíncrono que consulte en paralelo a tres modelos de lenguaje independientes. El sistema comparará semánticamente las respuestas calculando la similitud del coseno de sus embeddings; la decisión o transacción final solo procederá de forma automatizada si existe un consenso mayoritario (similitud >95%).
